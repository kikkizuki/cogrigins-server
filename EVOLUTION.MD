# Эволюция инфраструктуры проекта Cogrigins

Этот документ описывает путь развития сервера: от простых Bash-скриптов и облачных VPS до профессиональной контейнеризации и Self-hosting архитектуры.

---

## Этап 1: "Legacy" (Эпоха Screen)
*Использование локальных скриптов и терминальных сессий в облаке.*

```ascii
 [ Стек ] --> Bash + Screen + OpenJDK 17
 [ Хранение ] --> Локальные папки (/home/minecraft/Server)

Особенности этапа:

    Среда: Аренда облачного VPS с жестко ограниченными ресурсами.

    Управление: Ручной запуск через screen, требующий постоянного контроля активных сессий.

    Риски: Отсутствие изоляции процессов. Ошибки в скриптах приводили к переполнению диска логами или зависанию всей системы.

    Производительность: Низкий TPS и высокий MSPT из-за слабой однопоточной производительности облачных процессоров.

Этап 2: "Self-hosting" (Docker & Собственное железо)

Текущая архитектура, ориентированная на максимальную стабильность и FPS.
Фрагмент кода

 [ Железо ] --> Intel Core i5-12400 (6/12)
 [ Стек ]   --> Docker + Docker Compose + OpenJDK 17
 [ Сеть ]   --> Fast Reverse Proxy (FRP)
 [ RAM ]    --> 12GB Limit (10GB Heap)

Ключевые изменения:

    Infrastructure as Code: Вся конфигурация описана в docker-compose.yml. Развертывание среды теперь занимает секунды.

    Сетевой шлюз (FRP): Использование Fast Reverse Proxy позволило вынести точку входа на внешний VPS, скрыв домашний IP и защитив локальную сеть.

    Graceful Shutdown: Скрипт backup_docker.sh корректно останавливает контейнер. Это гарантирует сброс кэша чанков на диск перед архивацией.

    Оптимизация G1GC: Внедрение флагов Aikar's для минимизации задержек (GC pauses) при работе с тяжелыми модами вроде Create.

Извлеченные уроки (Post-Mortem)

    Стандартизация: Опыт со старыми скриптами научил использовать исключительно UTF-8, чтобы избежать повреждения комментариев (те самые "ромбики" в архивных файлах).

    Контроль ресурсов: Жесткие лимиты памяти в Docker предотвращают падение хост-системы при критических утечках RAM в Java-машине.

    Безопасность данных: Прямое копирование (cp) активных файлов мира признано небезопасным и заменено на архивацию остановленного инстанса.
